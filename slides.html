Overview
  • R&D project focuses on medical image processing software - in particular its limitations

  • Discusses DICOM image format, the difficulties in processing it and the necessity of converting it

  • Addresses inability to directly load a single slice without having to load an entire 3D image

  • Compares and contrasts a variety of medical image processors and viewers
3D Medical Images
  • Created from MRIs, CTs, X-Rays, etc.

  • Differ from traditional images in a number of ways
- Three unique planes
- Provide additional metadata
DICOM 
  • "Digital Imaging and Communications in Medicine"

  • Consists of file format and networking protocol

  • Used in all modern medical imaging systems

  • Accessed through DICOM Viewers

  • Contain comprehensive metadata

  • Is an interface definition
DICOM Data Model
  • Text-based tagged file format

  • Creates attribute/pairs, has accompanying .raw file

  • MetaImage header file that points to data

  • Very easy to read and parse .mhd files 
           (DICOM much more difficult - binary & requires
             non-trivial reading of bytes)

  • Much less widespread than DICOM
MHD (Meta Header)

ObjectType = Image
NDims = 3
DimSize = 256 256 64
ElementType = MET_USHORT
ElementDataFile = image.raw
example.mhd
Reading DICOM Images

  • One file per slice

  • Series of slices required for one 3D image

  • Slices can be scattered across various directories and 
subdirectories, with inconsistent file names
and Tag values

  • This makes it difficult to read an entire DICOM 3D images

Desk Web Interface
  • Browse, annotate, process and visualise 3D medical models

  • Slicing, segmenting and 3D rendering

  • Can process large quantities of data 

  • Processes on server-side

  • Browser caching

  • Easily accessible, no installation
Desk Web Interface
  • no direct processing of DICOM files

  • No processing of single image slices (out of a series)

  • Can become strained while processing larger 3D images

  • Rendering still on client

On-the-fly processing of .mhd files
DEMO
Advantages
  • no more need to load whole image into memory

  • saving a lot of processor power, as resources only used on request

  • As a result of that two points: Much faster displaying of large images (at least to have a first look at it)
Limitations
  • just one direction of image displaying possible (z - direction)

  • first look into image is very fast, but handling afterwards can lag (if displaying fast images)

  • just .mhd files with uncompressed RAW data are supportet
